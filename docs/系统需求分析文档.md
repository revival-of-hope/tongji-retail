# 商品零售管理系统 - 需求分析文档

## 1. 引言

### 1.1 编写目的
本文档旨在明确“商品零售管理系统”的功能需求、非功能需求及业务流程，为后续的系统设计、数据库设计及代码开发提供依据。

### 1.2 项目背景
随着电子商务的快速发展，传统的线下零售模式已无法满足现代消费者的购物需求。本项目旨在开发一套基于 B/S 架构的电商平台商品零售管理系统，支持多角色（管理员、商家、顾客、客服）协同工作，提供从商品展示、购物车、订单支付到售后评价的完整闭环服务。

## 2. 角色与用例分析

系统包含四类核心用户角色，各角色的主要用例（功能点）如下：

### 2.1 顾客（Customer）
- **浏览商品**：在首页通过瀑布流查看商品，支持按分类、价格、销量、评分进行筛选和排序。
- **商品搜索**：通过关键词搜索商品名称或描述。
- **购物车管理**：将商品加入购物车，修改数量，选中商品进行结算。
- **订单管理**：填写收货地址提交订单，模拟支付，查看订单状态（待支付、待发货、已发货、已完成、已取消），确认收货，取消未发货订单。
- **商品评价**：对已完成的订单商品进行星级评分和文字评价。
- **客服工单**：提交售后或咨询工单，查看客服回复。
- **申请入驻**：申请成为平台商家。

### 2.2 商家（Merchant）
- **商品管理**：发布新商品（需管理员审核），查看本店商品列表，修改库存和价格。
- **订单处理**：查看本店商品的销售订单，对已支付订单进行发货操作。
- **销售报表**：查看本店近 30 天的每日销售额折线图及商品销量排行榜。

### 2.3 管理员（Admin）
- **数据概览**：查看平台总销售额、总订单数、总用户数等核心指标，以及分类销售占比饼图。
- **商品审核**：审批商家发布的新商品，决定是否允许上架。
- **商家审批**：审批顾客提交的商家入驻申请。
- **用户与订单管理**：查看全平台所有用户和订单信息。

### 2.4 客服（Customer Service）
- **工单处理**：查看分配给自己的顾客工单，进行回复和状态更新（处理中、已解决）。

## 3. 核心业务流程

### 3.1 购物下单流程
1. 顾客浏览商品详情，选择数量加入购物车。
2. 顾客进入购物车，勾选需要结算的商品，点击“去结算”。
3. 系统验证库存是否充足。
4. 顾客填写收货地址，提交订单。系统生成唯一订单号，锁定当前商品价格（价格快照），并从购物车中移除对应商品。
5. 顾客在 30 分钟内完成支付。支付成功后，系统扣减实际库存，增加商品销量，订单状态变更为“待发货”。

### 3.2 商家入驻与商品发布流程
1. 顾客提交商家入驻申请（填写店铺名称和描述）。
2. 管理员审批通过后，该顾客角色升级为“商家”。
3. 商家在后台发布新商品，填写价格、库存、分类、图片等信息。
4. 新商品状态默认为“待审核”。
5. 管理员在后台审核通过后，商品状态变更为“已上架”，顾客即可在前台看到该商品。

## 4. 非功能需求

### 4.1 技术架构要求
- **架构模式**：B/S（浏览器/服务器）架构。
- **前端技术**：Next.js 14 (React) + Tailwind CSS，支持响应式布局和瀑布流展示。
- **后端技术**：C# ASP.NET Core 8 Web API。
- **数据库**：Oracle 21c XE。
- **ORM 框架**：Entity Framework Core。
- **部署方式**：Docker Compose 容器化一键部署。

### 4.2 性能与安全要求
- **并发处理**：下单扣减库存时需保证原子性，防止超卖。
- **数据安全**：用户密码使用 BCrypt 算法进行哈希加密存储。
- **认证授权**：使用 JWT (JSON Web Token) 进行无状态的身份认证和接口权限控制。
- **价格快照**：订单明细中必须记录下单时的商品单价，防止商家后续修改价格影响历史订单金额。
