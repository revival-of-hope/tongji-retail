# 商品零售管理系统 - 系统设计与实现文档

## 1. 系统架构设计

### 1.1 整体架构
本系统采用标准的 **B/S（Browser/Server）前后端分离架构**，分为表现层、业务逻辑层和数据访问层。
- **表现层（前端）**：使用 Next.js 14 (React) 构建，负责用户界面展示和交互，通过 RESTful API 与后端通信。
- **业务逻辑层（后端）**：使用 C# ASP.NET Core 8 Web API 构建，负责处理核心业务逻辑、权限校验和数据验证。
- **数据访问层（数据库）**：使用 Oracle 21c XE 存储数据，通过 Entity Framework Core (EF Core) 进行对象关系映射（ORM）。

### 1.2 部署架构
系统采用 **Docker Compose** 进行容器化编排，包含三个核心服务：
1. `oracle-db`：运行 Oracle 21c XE 数据库容器
2. `backend`：运行 ASP.NET Core 8 Web API 容器，通过内部网络连接 `oracle-db`。
3. `frontend`：运行 Next.js 14 前端容器，通过内部网络调用 `backend` 提供的 API 接口。

## 2. 核心模块设计与实现

### 2.1 认证与授权模块
- **实现方式**：基于 JWT (JSON Web Token) 的无状态认证。
- **流程**：
  1. 用户在前端输入用户名和密码。
  2. 后端使用 BCrypt 算法验证密码哈希。
  3. 验证通过后，后端生成包含 `UserId` 和 `Role` 信息的 JWT 字符串返回给前端。
  4. 前端将 JWT 存储在 Zustand 全局状态和 LocalStorage 中。
  5. 后续请求在 HTTP Header 中携带 `Authorization: Bearer <token>`。
  6. 后端通过 `[Authorize(Roles = "Admin")]` 等特性进行接口级别的权限拦截。

### 2.2 商品管理模块
- **瀑布流展示**：前端首页使用 CSS Grid 布局实现商品瀑布流展示，支持分页加载。
- **商品发布与审核**：
  - 商家在后台提交新商品信息，初始状态为“待审核”。
  - 管理员在后台审核通过后，状态变更为“已上架”。
  - 只有状态为“已上架”且库存大于 0 的商品才会展示在顾客端首页。

### 2.3 购物车与订单模块
- **购物车设计**：购物车数据持久化存储在数据库中，支持跨设备同步。
- **价格快照机制**：
  - 顾客下单时，系统会将当前商品的单价（`Product.Price`）复制到订单明细表（`OrderItem.UnitPrice`）中。
  - 即使商家后续修改了商品价格，历史订单的金额也不会受到影响，保证了财务数据的准确性。
- **库存扣减策略**：
  - 采用“支付后扣减库存”策略。
  - 顾客提交订单后，订单状态为“待支付”，此时不扣减实际库存。
  - 顾客完成支付后，系统在同一个数据库事务中更新订单状态为“待发货”，并扣减对应商品的实际库存（`StockQuantity`）和增加销量（`SoldCount`）。
  - 若支付时发现库存不足，则支付失败并提示用户。

### 2.4 数据报表模块
- **实现方式**：后端通过 EF Core 的 LINQ 查询和聚合函数（如 `GroupBy`, `Sum`, `Count`）生成统计数据。
- **图表展示**：前端使用 Recharts 库渲染数据可视化图表。
- **管理员报表**：包含平台总销售额、总订单数、近 30 天每日销售额折线图、各分类销售占比饼图。
- **商家报表**：包含本店近 30 天每日销售额折线图、本店商品销量排行榜 Top 10。

## 3. 接口设计规范

系统采用 RESTful API 设计风格，统一返回 JSON 格式数据。

### 3.1 统一响应格式
```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

### 3.2 核心 API 列表
- `POST /api/auth/login`：用户登录
- `POST /api/auth/register`：用户注册
- `GET /api/products`：获取商品列表（支持分页、分类筛选）
- `GET /api/products/{id}`：获取商品详情
- `POST /api/orders/cart`：加入购物车
- `POST /api/orders`：提交订单
- `POST /api/orders/pay`：模拟支付订单
- `GET /api/reports/overview`：获取管理员数据概览

## 4. 异常处理与日志
- **全局异常处理**：后端配置了全局异常处理中间件（Exception Middleware），捕获所有未处理的异常，统一返回 500 状态码和友好的错误提示，避免将敏感的堆栈信息暴露给前端。
- **前端错误提示**：前端使用 `react-hot-toast` 库，在捕获到 API 错误响应时，弹出友好的 Toast 提示框。
